# Cross-platform C++ SDK with a C-ABI boundary and XCFramework pipeline

## Why this shape

A native SDK meant to outlive one client app has one real design constraint:
**the public surface must be pure C**. C++ has no stable ABI across
compilers/versions, so any header that leaks `std::string`, templates,
exceptions, or virtual dispatch across the library boundary locks every
consumer to the exact toolchain that built the library. A pure C header
(`extern "C"`, POD structs, opaque handles, function-pointer callbacks) is
readable from Swift, Kotlin/JNI, C#, or plain C — so the same compiled core
serves an iOS/macOS app today and an Android `.aar` or Windows `.dll` later
without a rewrite. Everything in this skill exists in service of that one
constraint; the patterns below (Pimpl, interfaces, factory, callbacks) are
how you get to write normal, comfortable C++ *underneath* that boundary
without it leaking through.

Read `references/architecture-patterns.md` for the full rationale behind
each pattern before deviating from them — it explains the *why*, not just
the *what*, so you can make good judgment calls for the user's actual
domain. Read `references/build-pipeline.md` when touching CMakeLists.txt,
build_xcframework.sh, or module.modulemap — the build side has a few
non-obvious traps (module maps aren't generated by CMake automatically,
iOS cross-compiles need specific SDK/arch flags, etc.).

## Workflow

### 1. Interview the user

Don't guess these — ask, because they shape every generated file:

- **SDK name** (PascalCase, e.g. `SensorSDK`) — becomes the library target,
  the `.xcframework` name, and the umbrella header name.
- **Entity name** (PascalCase, e.g. `Sensor`, `Camera`, `Printer`) — the
  thing the SDK discovers and connects to. Drives `{Entity}Manager`,
  `{Entity}Session`, `{Entity}Info`.
- **Channel name** (PascalCase, default `Transport`; alternatives: `Link`,
  `Channel`) — the abstract backend interface, `I{Channel}`.
- **C API prefix** (short, uppercase, e.g. `SNS`) — prefixes every exported
  C function and type (`SNS_GetVersion`, `SNSDeviceInfo`).
- **Internal namespace** (lowercase, e.g. `sns`) — wraps the C++-only code
  below the boundary.
- **Bundle identifier** (default `com.example.<sdkname-lower>`), **author**
  (default from `git config`), **deployment targets** (default macOS 13.0 /
  iOS 16.0).
- **Output location** — a fresh repo, or alongside an existing Swift client
  project (mirroring a `<SDK>/` + `<SDK>.xcframework/` + `<Client>/` layout).

If the user already described their domain in the conversation (e.g. "BLE
heart-rate strap"), infer sensible defaults for entity/channel names and
confirm them rather than re-asking from scratch.

If a `SDK/Public/Internal/Private` three-tier split was explicitly
requested (see the "Layered structure" note in
`references/architecture-patterns.md`), mention that the generated skeleton
uses the flatter `include/` + `src/` split instead — it's the same
separation (pure-C public surface vs. free-to-use-STL internals) with less
directory ceremony. Offer to move files into `Public/Internal/Private` if
the user actually wants that stricter layout; don't do it unasked.

### 2. Generate the skeleton

Run the bundled scaffolder — it's deterministic and already
compile-verified, so prefer it over writing the files by hand:

```bash
python3 <skill-dir>/scripts/scaffold.py \
  --sdk-name SensorSDK --entity-name Sensor --channel-name Link \
  --api-prefix SNS --namespace sns \
  --output /path/to/repo
```

Run `python3 scripts/scaffold.py --help` for every flag (bundle-id, author,
year, macos-target, ios-target, force). It refuses to overwrite a non-empty
output directory unless `--force` is passed — respect that guard rather
than routing around it; if the target already has content, ask the user
before forcing.

This produces:

```
<repo>/
  <SDK>/
    include/<SDK>.h              C-ABI public header (extern "C", POD only)
    src/domain/types.h           internal C++ types (STL OK, never exported)
    src/transport/I<Channel>.h           pure abstract backend interface
    src/transport/Simulator<Channel>.*   concrete simulator backend
    src/transport/<Channel>Factory.*     single creation point for backends
    src/discovery/<Entity>Manager.*      Pimpl root service (discovery)
    src/session/<Entity>Session.*        Pimpl live-connection object
    src/capi/<entity>_c_api.cpp          C-ABI facade (POD <-> C++ conversion)
    src/version.cpp
    framework/module.modulemap   hand-maintained Clang module map
    tools/smoke_test.cpp         CLI sanity check, skipped when cross-compiling
    CMakeLists.txt
  build_xcframework.sh           builds 3 slices, wraps into <SDK>.xcframework
  CLAUDE.md                      build commands + architecture notes for this repo
  .gitignore
```

### 3. Verify it actually builds before handing it back

Never claim the scaffold works without running it — templates drift and a
placeholder substitution bug is easy to miss by eye. At minimum:

```bash
cmake -S <SDK> -B <SDK>/build-macos -DCMAKE_BUILD_TYPE=Release
cmake --build <SDK>/build-macos --target <sdk-lower>_smoke_test
./<SDK>/build-macos/<sdk-lower>_smoke_test   # expect "SMOKE PASSED"
```

If the user has Xcode command line tools and time to spare, also run
`./build_xcframework.sh` once to confirm all three slices (macOS,
iOS device, iOS simulator) configure and link — this is what actually
catches iOS-cross-compile-only breakage (the macOS-only build above won't).

### 4. Fill in the real domain

The scaffold ships with a deliberately generic placeholder: `Sample` (raw
byte buffer + timestamp) standing in for whatever payload the real device
produces (an image frame, a sensor reading, a telemetry packet). Point the
user at:

- `src/domain/types.h` and `include/<SDK>.h` — replace `Sample`/`{API}Sample`
  with the real payload shape, keeping the public struct POD with a leading
  `struct_size` if it's likely to grow.
- `src/capi/<entity>_c_api.cpp` — the conversion function (`toC`) is the one
  place that translates the new payload; every other C API function is a
  thin forward and rarely needs domain changes.
- `src/transport/Simulator<Channel>.cpp` — extend `run()` to fabricate
  *realistic failures* for this domain, not just happy-path data. The
  ultrasound project this pattern is drawn from simulates battery drain,
  overheating, Wi-Fi drops with reconnects, and frame jitter — because the
  simulator is what UI/app developers build and test against before
  hardware exists, so it has to lie convincingly about everything that can
  go wrong, not just emit clean samples forever.
- `src/transport/I<Channel>.h` and `<Channel>Factory.*` — add a second,
  hardware-backed implementation here when real hardware is available; nothing
  above the interface needs to change.

### 5. Swift client integration (if applicable)

If the user has (or wants) a Swift client:

- The client links `<SDK>.xcframework` and does `import <SDK>` — this only
  works because `framework/module.modulemap` was injected into the
  `.framework` bundle by the CMake post-build step. If `import` fails in
  Xcode, check the module map first (see `references/build-pipeline.md`).
- If the client project is XcodeGen-managed (`project.yml`), the xcframework
  is added as a `dependencies:` entry with `embed: true`, referenced by
  relative path (e.g. `../<SDK>.xcframework`), on every target that needs it.
- Opaque `*Ref` handles cross into Swift as `OpaquePointer`; POD structs cross
  as Swift structs; C function-pointer callbacks are bridged with `@convention(c)`
  closures or a small ObjC/Swift wrapper — mention this only if the user is
  actively writing the Swift call site, it's outside this skill's scope to
  generate that layer.

## Reviewing/retrofitting an existing SDK

If the user has an existing C++ library instead of a blank repo, don't run
the scaffolder over it. Instead, read their current headers and CMake setup,
then check them against `references/architecture-patterns.md` point by
point: is the public header pure C? Is there a Pimpl boundary? Do backends
sit behind an interface with a factory? Is there a module map and a
post-build injection step? Report gaps and offer to add the missing pieces
incrementally, reusing the relevant template files in `assets/` as a
reference for shape (rename by hand — the scaffolder itself assumes an
empty target directory).
